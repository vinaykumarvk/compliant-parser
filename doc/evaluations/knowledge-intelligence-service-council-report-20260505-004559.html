Knowledge Intelligence Service Council Report

Adversarial validation of the standalone KIS BRD, generated 2026-05-05.

Recommendation

Proceed with KIS as a standalone reusable service, but narrow the first release and strengthen security, privacy, versioning, and quality gates before implementation.

The council agreed that PS-WMS has valuable reusable patterns, but the BRD must explicitly add control-plane/data-plane separation, published knowledge snapshots, domain templates, embedding privacy, isolation tests, idempotency, legal hold, and migration adapters.

Council Agreement

Right Boundary

KIS should be an independent platform service rather than a feature inside compliant-parser.

Reusable Core

PS-WMS graph, wiki, hybrid retrieval, reasoning, and governance features should be generalized.

Release Blockers

Privacy, secret handling, domain isolation, audit, and provider governance must be non-negotiable.

Advisor Analyses

The Proponent

The BRD has the right product boundary and captures PS-WMS differentiators: ingestion, chunks, embeddings, vector/graph/fact/wiki retrieval, reasoning patterns, prompt governance, provider routing, cost tracking, redaction, shadow mode, output fatigue, and golden sets. It should proceed, but v1 should be anchored on a narrow compliant-parser integration path.

The Contrarian

The largest risks are overbroad scope, credential exposure, weak isolation, and privacy leakage through embeddings. The BRD should make secret handling, domain isolation, and privacy controls release blockers, and reduce the first release surface.

The First Principles Thinker

The real goal is operationally trustworthy knowledge, not just RAG. The BRD should separate control plane, data plane, retrieval contract, and consuming applications, and should add published knowledge snapshots with rollback.

The Outsider

The idea is valuable but intimidating. Domain admins need templates and guided setup so they can create useful knowledge bases without understanding every low-level graph, vector, wiki, prompt, and provider setting.

The Executor

The fastest path is to extract reusable PS-WMS contracts and tests, build a small KIS core, and integrate compliant-parser first. Add idempotency before retries create duplicate documents, edges, wiki articles, or external LLM charges.
